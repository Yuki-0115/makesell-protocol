## 1. プロダクト概要

| 項目 | 内容 |
|------|------|
| **プロダクト名** | MakeSell Protocol |
| **一言で言うと** | 作る人と売る人をつなぐ契約基盤 |
| **誰の** | プロダクトを作れるが売れないインディー開発者 / 売る力はあるが商品がない営業人材・インフルエンサー |
| **何を解決する** | 「作った人が売る」という役割の強制結合を分離し、才能のミスマッチによる機会損失をなくす |

### コンセプト

> **AI時代、才能は「混ぜる」より「分けた方が伸びる」**

## 2. 市場背景：なぜ今このプラットフォームが必要か

### AI開発ツールの爆発的普及

| ツール | ユーザー規模 | 特徴 |
|--------|-------------|------|
| **Replit** | 3,000〜4,000万人（MAU 1,800万） | 学習〜プロトタイプ〜社内ツール、法人50万社以上 |
| **Lovable** | 数百万人（1日10万プロジェクト生成） | 8ヶ月で$100M ARR → $200M ARRへ倍増、評価額$6.6B |
| **Bolt.new** | 急成長中 | ワンクリックデプロイ |
| **Cursor** | 開発者に浸透中 | AIペアプログラミング |

### この流れが生む構造的問題

```
「作れる人」が爆発的に増える
        ↓
「売れる人」は増えない
        ↓
プロダクトの墓場が量産される
        ↓
才能のミスマッチが拡大
```

### MakeSellの立ち位置

> **Replit 4,000万人、Lovable 1日10万プロジェクト——**
> **この時代に足りないのは「作る力」ではなく「届ける力」。**

| 従来の構造 | MakeSellが実現する構造 |
|-----------|----------------------|
| 作る人 = 売る人（強制結合） | 作る人 ≠ 売る人（分離可能） |
| 営業できない開発者 → 放置 | 営業できない開発者 → セラーに委託 |
| 商材がない営業 → 他社依存 | 商材がない営業 → ビルダーと直接契約 |

**MakeSellは、AI開発時代の「才能の非対称性」を接続するインフラになる。**

### グローバル戦略：初期から世界を見る

**先行事例の教訓：**

| 企業 | 拠点 | 戦略 |
|------|------|------|
| **Lovable** | スウェーデン（ストックホルム） | 初期からグローバル展開。ユーザーは欧州・米国・アジア・南米で均等分布 |
| **Replit** | 米国 | 40カ国以上で利用、教育から法人まで |

> **LovableのCEO Anton Osika：**
> 「投資家からシリコンバレーへの移転を勧められたが断った。スウェーデンからグローバルAI企業を作れることを証明した」

**MakeSellの方針：**

- **言語：** 英語ファースト（日本語は同時対応）
- **決済：** Stripe Connect（グローバル対応済み）
- **契約：** 国際対応可能な設計
- **ターゲット：** Vibe Codingで生まれるプロダクトは国境を越える。MakeSellも同様。

> **「日本発、初日からグローバル」**
> ローカライズは後回しにしない。世界中のビルダーとセラーを接続する。

## 4. ユーザーストーリー

### 優先度順

| # | 誰が | 何をしたい | なぜ |
|---|------|-----------|------|
| 1 | ビルダーとして | 作ったプロダクトを登録して、売ってくれる人を見つけたい | 自分で営業する時間も才能もないから |
| 2 | セラーとして | 売れそうなプロダクトを探して、販売権を得たい | 自分の営業力を活かせる商材が欲しいから |
| 3 | ビルダーとして | 売上の分配を自動で受け取りたい | 請求・回収の手間を省きたいから |
| 4 | セラーとして | 販売実績を可視化したい | 自分の成果を証明して、より良い案件を獲得したいから |
| 5 | プラットフォームとして | トラブル時の責任分界点を明確にしたい | 感情的な揉め事を契約で防ぎたいから |

## 5. 機能一覧（MoSCoW法）

### Must（MVP必須）

| # | 機能 | 詳細 |
|---|------|------|
| 1 | プロダクト登録 | 名前・説明・価格・分配率 + **販売形態選択（独占/非独占）** |
| 2 | セラー申請・マッチング | 手動承認。申請時必須：過去の販売実績 or 得意業界 + **ポートフォリオ（実績リンク）公開** |
| 3 | 売上記録・分配計算 | **トリガー定義あり**（下記参照） |

### 独占 / 非独占の設計

| 販売形態 | 内容 | ビルダーのメリット | セラーのメリット |
|----------|------|-------------------|-----------------|
| **独占** | 1セラーのみに販売権付与 | 責任明確、ブランド管理しやすい | 競合なし、本気で売れる |
| **非独占** | 複数セラーが同時に販売可 | 拡散力最大化 | 参入ハードル低い |

**登録時の選択肢：**
- 独占（期間：90日 / 180日 / 無期限）
- 非独占（上限人数：無制限 / 5人まで / 10人まで）

### 売上記録トリガー

| トリガー | 処理 |
|----------|------|
| Stripe Webhook | 自動で売上記録 → 分配計算 → Stripe Connectで自動分配 |

> **設計思想：** 初期から自動化。人力オペレーションはスケールしない。

### 入力ガイド（UI上で表示）

```
【自己紹介文のコツ】
・何が得意か（1行）
・過去に何をやったか（1〜2行）
・どんな人と組みたいか（1行）
→ これで50〜100文字。十分です。

【プロダクト説明のコツ】
・誰向けか（1行）
・何ができるか（2〜3行）
・他との違い（1行）
→ これで100〜200文字。十分です。
```

### 自動表示される実績（信頼スコア）

| 指標 | 対象 | 内容 |
|------|------|------|
| 登録日 | 両方 | いつから使ってるか |
| 累計売上 | 両方 | 金額 or ランク表示（A/B/C） |
| 契約継続率 | 両方 | 途中解除されてないか |
| 通報数 | 運営のみ閲覧 | 問題行動の検知用 |

### 信頼構造の設計思想

> **審査しない。でも「隠せない」構造にする。**

```
検査しない
  ↓
全部開示させる
  ↓
情報不足 → 選ばれない → 淘汰
  ↓
嘘 → 通報 → BAN
  ↓
残るのは信頼できる人だけ
```

### リリース後1ヶ月の目標

| 指標 | 目標値 | 現実性 |

## 8. リスク・前提条件

### 前提条件

- 初期プロダクトは我々自身のプロダクトを使用
- セラーはSNS募集で手動審査
- 決済はStripe or 外部リンク（プラットフォーム内決済は後回し）

### 依存関係

- Stripeアカウント（分配用）
- 契約雛形の完成（責任分界点の明文化）

### リスク一覧

| リスク | 影響度 | 対策 |
|--------|--------|------|
| セラーが売れない・動かない | 高 | ①成果ゼロ60日で自動解除 ②申請時に過去実績必須 ③初期は手動スクリーニング |
| 販売表現でトラブル | 中 | NG表現リスト＋責任はセラー明記 |
| ビルダーがプロダクト放置 | 中 | サポート期限を契約に明記 |
| 分配でもめる | 高 | 決済ログ全件記録、Stripe Connect事前テスト |

## 9. 責任分界点

### 明確な切り分け

| 責任領域 | 担当 |
|----------|------|
| プロダクト品質・バグ | ビルダー |
| 販売表現・誇張・炎上 | セラー |
| 決済・分配・ログ | MakeSell |

### MakeSellの立ち位置

> MakeSellは「販売代理店」ではない。
> **「契約実行レイヤー」** である。

### プロダクト審査に関する免責

> MakeSellは、掲載されるプロダクトの品質・機能・適法性について
> 審査・検査・保証を一切行いません。
> 掲載は形式要件を満たした時点で自動的に行われ、
> 内容についてはすべてビルダーが責任を負います。
> セラーは、販売前にプロダクトを自身で確認する責任を負います。

## 12. 販売委託契約書（雛形 v1.1）

### 販売委託契約書（MakeSell Protocol）

本契約は、
プロダクト提供者（以下「ビルダー」） と
販売協力者（以下「セラー」） との間で、
MakeSell Protocol（以下「本プラットフォーム」）を介して行われる
販売協力に関し、以下のとおり締結される。

---

#### 第1条（契約の目的）

本契約は、ビルダーが開発したプロダクトについて、
セラーが自己の裁量と責任において販売活動を行い、
その対価として売上を分配する条件を定めることを目的とする。

---

#### 第2条（契約形態）

1. 本契約は雇用契約・代理店契約・業務委託契約には該当しない。
2. セラーは、ビルダーまたはMakeSellの代理人・代表者ではない。
3. セラーは、自己の判断と責任において販売活動を行うものとする。

> ※ ここが「販売代理店ではない」根拠条文。

---

#### 第3条（販売形態）

販売形態は、以下のいずれかとする。

| 形態 | 内容 |
|------|------|
| 独占販売 | 期間：90日 / 180日 / 365日（選択） |
| 非独占販売 | 上限人数：3人 / 5人 / 10人 / 無制限（選択） |

販売形態および期間は、MakeSell上の登録情報をもって確定する。

---

#### 第4条（売上および分配）

1. 売上は、顧客が支払いを完了した時点で発生する。
2. 売上の分配率は、MakeSell上で登録された内容に従う。
3. 分配は、決済完了後、Stripe Connectを通じて自動実行される。

---

#### 第5条（販売活動の範囲）

1. セラーは、自己の責任においてプロダクトの販売促進を行う。
2. セラーは、以下の行為を行ってはならない：
   - ビルダーが定めたNG表現を使用した販売活動
   - プロダクトの機能・効果について虚偽の説明
   - 法令に違反する販売手法

---

#### 第6条（禁止事項）

以下の行為を禁止する：

1. 虚偽の実績・経歴の申告
2. プロダクトの無断改変・再販売
3. 顧客情報の不正利用
4. MakeSellの信用を毀損する行為

---

#### 第7条（契約期間と終了）

1. 本契約は、MakeSell上でマッチングが成立した時点で発効する。
2. 以下の場合、契約は終了する：
   - 独占契約の期間満了
   - いずれかの当事者による解除申請（30日前通知）
   - 成果ゼロ状態が60日継続（自動解除）
   - 禁止事項違反によるMakeSellからの強制解除

---

#### 第8条（免責）

1. MakeSellは、プロダクトの品質・適法性について一切保証しない。
2. MakeSellは、セラーの販売活動に起因するトラブルについて責任を負わない。
3. ビルダー・セラー間の紛争は、当事者間で解決するものとする。

#### 第9条（準拠法・管轄）

本契約は日本法に準拠し、紛争が生じた場合は東京地方裁判所を第一審の専属的合意管轄裁判所とする。

## 13. リーガルチェック結果

### 結論

> 現行設計は「かなり安全寄り」。日本で違法プラットフォームになる可能性は低い。

### 主要論点と対応状況

| 論点 | 状態 | 対応 |
|------|------|------|
| 販売代理店該当性 | ○ | 第2条で明確に否定 |
| 特定商取引法 | ○ | 販売事業者情報表示で対応 |
| 決済・資金移動業 | ○ | Stripe Connect利用で回避 |
| 景品表示法 | ○ | NG表現明記で回避 |
| 著作権 | ○ | ビルダー責任明記 |
| 労働法 | ○ | 雇用性否定条文で回避 |

### 絶対にやってはいけないこと

| NG行為 | 理由 |

| MakeSellがCSを引き受ける | 当事者性が発生 |
| 成果保証・収益保証を言う | 詐欺・景表法リスク |
| 自前ウォレット・一時預かり | 資金移動業に該当 |

## 14. 直接取引ポリシー

### 設計思想

> **「禁止しない。ただし、通した方が得」な構造を作る**

| 事実 | 対応 |
|------|------|
| 直接連絡は防げない | 禁止しても抜けられる。監視コストも高い |
| 禁止するとUX悪化 | ユーザーが離れる |
| 他プラットフォームも同じ | ココナラ、クラウドワークス、Airbnb全部抱えてる |

### 方針：許容＋インセンティブ設計

| 取引形態 | MakeSellの対応 |
|----------|---------------|
| プラットフォーム経由 | 記録・分配・実績反映すべて対応 |
| 直接取引 | 関与しない。実績も反映しない |

### MakeSellを通すメリット（ビルダー側）

| メリット | 内容 |
|----------|------|
| 売上記録が残る | 実績として可視化→次のセラー獲得に有利 |
| トラブル時の証拠 | 決済ログ・契約ログが残る |
| 自動分配 | 請求・回収の手間ゼロ |
| 信頼スコア蓄積 | 継続率・売上がプロフィールに反映 |

### MakeSellを通すメリット（セラー側）

| メリット | 内容 |
|----------|------|
| 実績が公開される | 次の案件獲得に有利 |
| 未払いリスクなし | 決済完了後に分配される |
| 契約が明確 | 責任範囲が文書化されてる |
| セラーランク | 実績積むと優先マッチング（将来） |

### 直接取引のリスク（両者に伝える）

| リスク | 内容 |
|--------|------|
| 実績が残らない | 次のマッチングに活かせない |
| トラブル時に証拠なし | 「言った言わない」になる |
| 未払い・持ち逃げ | MakeSellは関与しない |
| 契約不備 | 責任分界が曖昧→揉める |

### 規約への記載文

```
【直接取引について】

MakeSellは、ビルダーとセラーが本プラットフォーム外で
直接連絡・取引を行うことを禁止しません。

ただし、プラットフォーム外での取引については、
MakeSellは一切関与せず、いかなる責任も負いません。

プラットフォーム経由の取引には、以下のメリットがあります：
・売上記録・実績の可視化
・契約条件の明文化
・決済ログによるトラブル防止
・自動分配による請求負担軽減
```

### 将来の拡張案（手数料差別化）

| 取引形態 | MakeSell手数料 | 実績反映 |
|----------|---------------|----------|
| プラットフォーム経由 | 10% | あり |
| 直接取引（申告制） | 0% | なし |

→ **「手数料払ってでも実績残したい」人がプラットフォームを使う構造**

## 16. 分配率設計

### 分配プラン（3段階固定）

| プラン名 | セラー取り分 | ビルダー取り分 | MakeSell | 想定ケース |
|----------|--------------|----------------|----------|------------|
| **ライト** | 20% | 70% | 10% | 紹介程度、既存顧客への案内 |
| **スタンダード** | 30% | 60% | 10% | 通常の営業活動（推奨デフォルト） |
| **プレミアム** | 40% | 50% | 10% | フルコミット、独占販売 |

### 具体例（月額¥5,000のSaaSが売れた場合）

| プラン | セラー | ビルダー | MakeSell |
|--------|--------|----------|----------|
| ライト | ¥1,000 | ¥3,500 | ¥500 |
| スタンダード | ¥1,500 | ¥3,000 | ¥500 |
| プレミアム | ¥2,000 | ¥2,500 | ¥500 |

### プラン選択ガイド（ビルダー向け）

```
【ライト（セラー20%）を選ぶ場合】
・セラーは「紹介するだけ」でOK
・顧客対応・サポートはビルダーが全部やる
・複数セラーに同時依頼したい（非独占）

【スタンダード（セラー30%）を選ぶ場合】
・セラーに積極的な営業活動を期待
・一次問い合わせ対応はセラーに任せたい
・迷ったらこれ（推奨デフォルト）

【プレミアム（セラー40%）を選ぶ場合】
・独占販売権を付与する
・セラーにフルコミットしてほしい
・高単価商品（月額1万円以上）向け
```

### 独占/非独占との組み合わせ推奨

| 販売形態 | 推奨プラン | 備考 |
|----------|------------|------|
| 非独占（複数セラー可） | ライト〜スタンダード | 競争原理が働く |
| 独占（1セラーのみ） | スタンダード〜プレミアム | セラーのコミット確保 |

> **注意：独占 + ライトはNG**（セラーのモチベーションが出ない）

### 継続課金（サブスク）の分配ルール

| 項目 | 設定 |
|------|------|
| **分配方式** | 継続分配（顧客が継続する限り毎月分配） |
| **分配期間** | 無期限（解約まで） |
| **解約時** | 解約月の分配で終了 |

**継続分配を採用する理由：**
- セラーが「売って終わり」にならない
- 顧客フォローへのインセンティブになる
- LTV意識の質の高い営業になる

## 18. 返金リスク対策

### リスクシナリオ

```
顧客が¥10,000を支払い
    ↓
自動分配完了（ビルダー・セラーが即出金）
    ↓
後日、顧客が返金要求
    ↓
ビルダー・セラーの残高ゼロ
    ↓
回収不能リスク発生
```

### 対策①：入金保留期間

| 項目 | 設定 |
|------|------|
| **保留期間** | 7日間 |
| **仕組み** | Stripeの`payout_schedule`で制御 |
| **入金開始** | 決済完了から8日目以降 |

### 対策②：返金ポリシーの制限

**顧客向け返金ポリシー：**

```
【返金ポリシー】

・購入後7日以内：全額返金可能
・購入後8日以降：原則返金不可
・月額課金：当月分の返金不可、次月から解約

※返金申請はMakeSellサポートまでご連絡ください
```

| 商品タイプ | 返金可能期間 | 返金条件 |
|------------|--------------|----------|
| 買い切り | 7日以内 | 未使用の場合 |
| 月額課金 | 原則不可 | 次月から解約対応 |
| 年額課金 | 7日以内 | 未使用の場合のみ全額 |

### 対策③：返金時の負担ルール

| 返金原因 | 負担者 | 備考 |
|----------|--------|------|
| ビルダー起因（バグ、機能不備、サービス停止） | **ビルダー全額** | プロダクト責任 |
| セラー起因（誇大広告、ミスマッチ誘導） | **セラー全額** | 販売責任 |
| 顧客都合（気が変わった等） | **按分負担** | 分配率に応じて |
| 不正利用・チャージバック | **按分負担** | 分配率に応じて |

> **MakeSellは返金費用を負担しない**（プラットフォーム免責）

### 対策④：ネガティブバランス対応

Stripe Connectの自動相殺機能を有効化：

```
セラーの次回売上 ¥5,000
    ↓
前回返金の未回収分 ¥3,000 を自動相殺
    ↓
実際の入金 ¥2,000
```

| 項目 | 設定 |
|------|------|
| **自動相殺** | 有効 |
| **相殺上限** | 次回売上の100%まで |
| **回収不能時** | MakeSellから督促 → 最終的にアカウント停止 |

### 返金フロー図

```
┌─────────────────────────────────────────────────────┐
│  顧客から返金リクエスト                              │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│  返金可否判定                                        │
│  ├→ 7日以内？ → 返金処理へ                          │
│  └→ 7日超過？ → 原則拒否（個別対応）                │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│  返金原因の特定                                      │
│  ├→ ビルダー起因 → ビルダー負担                     │
│  ├→ セラー起因 → セラー負担                         │
│  └→ 顧客都合 → 按分負担                             │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│  残高から自動控除                                    │
│  ├→ 残高あり → 即時控除・返金実行                   │
│  └→ 残高不足 → 次回売上から相殺                     │
└─────────────────────────────────────────────────────┘
```

### 契約書追記条項

```
【第○条（返金時の負担）】

1. 顧客から返金要求があった場合、以下のルールに従い処理する。

2. 返金原因の負担区分
   (1) ビルダー起因（プロダクトの瑕疵、機能不備、サービス停止等）
       → ビルダーが返金額の全額を負担する
   (2) セラー起因（誇大広告、虚偽説明、不適切な販売行為等）
       → セラーが返金額の全額を負担する
   (3) 顧客都合（単純な気変わり、誤購入等）
       → ビルダー・セラーが本契約の分配率に応じて按分負担する

3. MakeSellは、いかなる場合も返金費用を負担しない。

4. 返金発生時、MakeSellは負担者のStripeアカウント残高から
   返金相当額を自動控除する権利を有する。

5. 残高不足の場合、次回以降の売上から相殺する。
   相殺完了まで、当該アカウントへの入金を保留できる。

6. 相殺による回収が3ヶ月以上不可能な場合、
   MakeSellは当該アカウントを停止し、
   別途請求を行う権利を有する。
```

### 返金率の想定

| 業種 | 一般的な返金率 | MakeSell想定 |
|------|----------------|--------------|
| SaaS（月額） | 1〜3% | 2%以下 |
| デジタル商品 | 3〜5% | - |
| 物販 | 5〜10% | - |

> **SaaS中心のため、返金リスクは限定的と想定**
