## 1. プロダクト概要

| 項目 | 内容 |
|------|------|
| **プロダクト名** | MakeSell Protocol |
| **一言で言うと** | 作る人と売る人をつなぐ契約基盤 |
| **誰の** | プロダクトを作れるが売れないインディー開発者 / 売る力はあるが商品がない営業人材・インフルエンサー |
| **何を解決する** | 「作った人が売る」という役割の強制結合を分離し、才能のミスマッチによる機会損失をなくす |

### コンセプト

> **AI時代、才能は「混ぜる」より「分けた方が伸びる」**

---

## 2. 市場背景：なぜ今このプラットフォームが必要か

### AI開発ツールの爆発的普及

| ツール | ユーザー規模 | 特徴 |
|--------|-------------|------|
| **Replit** | 3,000〜4,000万人（MAU 1,800万） | 学習〜プロトタイプ〜社内ツール、法人50万社以上 |
| **Lovable** | 数百万人（1日10万プロジェクト生成） | 8ヶ月で$100M ARR → $200M ARRへ倍増、評価額$6.6B |
| **Bolt.new** | 急成長中 | ワンクリックデプロイ |
| **Cursor** | 開発者に浸透中 | AIペアプログラミング |

### この流れが生む構造的問題

```
「作れる人」が爆発的に増える
        ↓
「売れる人」は増えない
        ↓
プロダクトの墓場が量産される
        ↓
才能のミスマッチが拡大
```

### MakeSellの立ち位置

> **Replit 4,000万人、Lovable 1日10万プロジェクト——**
> **この時代に足りないのは「作る力」ではなく「届ける力」。**

| 従来の構造 | MakeSellが実現する構造 |
|-----------|----------------------|
| 作る人 = 売る人（強制結合） | 作る人 ≠ 売る人（分離可能） |
| 営業できない開発者 → 放置 | 営業できない開発者 → セラーに委託 |
| 商材がない営業 → 他社依存 | 商材がない営業 → ビルダーと直接契約 |

**MakeSellは、AI開発時代の「才能の非対称性」を接続するインフラになる。**

### グローバル戦略：初期から世界を見る

**先行事例の教訓：**

| 企業 | 拠点 | 戦略 |
|------|------|------|
| **Lovable** | スウェーデン（ストックホルム） | 初期からグローバル展開。ユーザーは欧州・米国・アジア・南米で均等分布 |
| **Replit** | 米国 | 40カ国以上で利用、教育から法人まで |

> **LovableのCEO Anton Osika：**
> 「投資家からシリコンバレーへの移転を勧められたが断った。スウェーデンからグローバルAI企業を作れることを証明した」

**MakeSellの方針：**

- **言語：** 英語ファースト（日本語は同時対応）
- **決済：** Stripe Connect（グローバル対応済み）
- **契約：** 国際対応可能な設計
- **ターゲット：** Vibe Codingで生まれるプロダクトは国境を越える。MakeSellも同様。

> **「日本発、初日からグローバル」**
> ローカライズは後回しにしない。世界中のビルダーとセラーを接続する。

### 3.4 MakeSellが解決する共通課題

| 問題 | ビルダーの課題 | セラーの課題 | MakeSellの解決策 |
|------|---------------|-------------|-----------------|
| **アクセス問題** | 市場・顧客にリーチできない | 良い商材にアクセスできない | マッチングで相互補完 |
| **信頼問題** | 知名度ゼロ、実績証明できない | 実績を可視化できない | プラットフォーム内実績表示 |
| **時間問題** | 作れるけど売る時間がない | 売れるけど商材探す時間がない | 役割分離で専念可能 |
| **収益構造問題** | 受託に逃げて自分のプロダクトが後回し | 単発案件ばかりで継続収益がない | 継続分配で資産型収入に転換 |

## 4. ユーザーストーリー

### 優先度順

| # | 誰が | 何をしたい | なぜ |
|---|------|-----------|------|
| 1 | ビルダーとして | 作ったプロダクトを登録して、売ってくれる人を見つけたい | 自分で営業する時間も才能もないから |
| 2 | セラーとして | 売れそうなプロダクトを探して、販売権を得たい | 自分の営業力を活かせる商材が欲しいから |
| 3 | ビルダーとして | 売上の分配を自動で受け取りたい | 請求・回収の手間を省きたいから |
| 4 | セラーとして | 販売実績を可視化したい | 自分の成果を証明して、より良い案件を獲得したいから |
| 5 | プラットフォームとして | トラブル時の責任分界点を明確にしたい | 感情的な揉め事を契約で防ぎたいから |

## 5. 機能一覧（MoSCoW法）

### Must（MVP必須）

| # | 機能 | 詳細 |
|---|------|------|
| 1 | プロダクト登録 | 名前・説明・価格・分配率 + **販売形態選択（独占/非独占）** |
| 2 | セラー申請・マッチング | ビルダーによる承認。申請時必須：過去の販売実績 or 得意業界 + **ポートフォリオ（実績リンク）公開** |
| 3 | 売上記録・分配計算 | **トリガー定義あり**（下記参照） |

### 独占 / 非独占の設計

| 販売形態 | 内容 | ビルダーのメリット | セラーのメリット |
|----------|------|-------------------|-----------------|
| **独占** | 1セラーのみに販売権付与 | 責任明確、ブランド管理しやすい | 競合なし、本気で売れる |
| **非独占** | 複数セラーが同時に販売可 | 拡散力最大化 | 参入ハードル低い |

**登録時の選択肢：**
- 独占（期間：90日 / 180日 / 365日 / 無期限）
- 非独占（上限人数：無制限 / 5人まで / 10人まで）

### 売上記録トリガー

| トリガー | 処理 |
|----------|------|
| Stripe Webhook | 自動で売上記録 → 分配計算 → Stripe Connectで自動分配 |

> **設計思想：** 初期から自動化。人力オペレーションはスケールしない。

### 入力ガイド（UI上で表示）

```
【自己紹介文のコツ】
・何が得意か（1行）
・過去に何をやったか（1〜2行）
・どんな人と組みたいか（1行）
→ これで50〜100文字。十分です。

【プロダクト説明のコツ】
・誰向けか（1行）
・何ができるか（2〜3行）
・他との違い（1行）
→ これで100〜200文字。十分です。
```

### 自動表示される実績（信頼スコア）

| 指標 | 対象 | 内容 |
|------|------|------|
| 登録日 | 両方 | いつから使ってるか |
| 累計売上 | 両方 | 金額 or ランク表示（A/B/C） |
| 契約継続率 | 両方 | 途中解除されてないか |
| 通報数 | 運営のみ閲覧 | 問題行動の検知用 |

### 信頼構造の設計思想

> **審査しない。でも「隠せない」構造にする。**

```
検査しない
  ↓
全部開示させる
  ↓
情報不足 → 選ばれない → 淘汰
  ↓
嘘 → 通報 → BAN
  ↓
残るのは信頼できる人だけ
```

## 12. 販売委託契約書（雛形 v1.1）

### 販売委託契約書（MakeSell Protocol）

本契約は、
プロダクト提供者（以下「ビルダー」） と
販売協力者（以下「セラー」） との間で、
MakeSell Protocol（以下「本プラットフォーム」）を介して行われる
販売協力に関し、以下のとおり締結される。

---

#### 第1条（契約の目的）

本契約は、ビルダーが開発したプロダクトについて、
セラーが自己の裁量と責任において販売活動を行い、
その対価として売上を分配する条件を定めることを目的とする。

---

#### 第2条（契約形態）

1. 本契約は雇用契約・代理店契約・業務委託契約には該当しない。
2. セラーは、ビルダーまたはMakeSellの代理人・代表者ではない。
3. セラーは、自己の判断と責任において販売活動を行うものとする。

> ※ ここが「販売代理店ではない」根拠条文。

---

#### 第3条（販売形態）

販売形態は、以下のいずれかとする。

| 形態 | 内容 |
|------|------|
| 独占販売 | 期間：90日 / 180日 / 365日 / 無期限（選択） |
| 非独占販売 | 上限人数：3人 / 5人 / 10人 / 無制限（選択） |

販売形態および期間は、MakeSell上の登録情報をもって確定する。

---

#### 第4条（売上および分配）

1. 売上は、顧客が支払いを完了した時点で発生する。
2. 売上の分配率は、MakeSell上で登録された内容に従う。
3. 分配は、決済完了後、Stripe Connectを通じて自動実行される。

---

#### 第5条（販売活動の範囲）

1. セラーは、自己の責任においてプロダクトの販売促進を行う。
2. セラーは、以下の行為を行ってはならない：
   - ビルダーが定めたNG表現を使用した販売活動
   - プロダクトの機能・効果について虚偽の説明
   - 法令に違反する販売手法

---

#### 第6条（禁止事項）

以下の行為を禁止する：

1. 虚偽の実績・経歴の申告
2. プロダクトの無断改変・再販売
3. 顧客情報の不正利用
4. MakeSellの信用を毀損する行為

---

#### 第7条（契約期間と終了）

1. 本契約は、MakeSell上でマッチングが成立した時点で発効する。
2. 以下の場合、契約は終了する：
   - 独占契約の期間満了
   - いずれかの当事者による解除申請（30日前通知）
   - 成果ゼロ状態が60日継続（自動解除）
   - 禁止事項違反によるMakeSellからの強制解除

---

#### 第8条（免責）

1. MakeSellは、プロダクトの品質・適法性について一切保証しない。
2. MakeSellは、セラーの販売活動に起因するトラブルについて責任を負わない。
3. ビルダー・セラー間の紛争は、当事者間で解決するものとする。

---

#### 第9条（準拠法・管轄）

本契約は日本法に準拠し、紛争が生じた場合は東京地方裁判所を第一審の専属的合意管轄裁判所とする。

## 14. 直接取引ポリシー

### 設計思想

> **「禁止しない。ただし、通した方が得」な構造を作る**

| 事実 | 対応 |
|------|------|
| 直接連絡は防げない | 禁止しても抜けられる。監視コストも高い |
| 禁止するとUX悪化 | ユーザーが離れる |
| 他プラットフォームも同じ | ココナラ、クラウドワークス、Airbnb全部抱えてる |

### 方針：許容＋インセンティブ設計

| 取引形態 | MakeSellの対応 |
|----------|---------------|
| プラットフォーム経由 | 記録・分配・実績反映すべて対応 |
| 直接取引 | 関与しない。実績も反映しない |

### MakeSellを通すメリット（ビルダー側）

| メリット | 内容 |
|----------|------|
| 売上記録が残る | 実績として可視化→次のセラー獲得に有利 |
| トラブル時の証拠 | 決済ログ・契約ログが残る |
| 自動分配 | 請求・回収の手間ゼロ |
| 信頼スコア蓄積 | 継続率・売上がプロフィールに反映 |

### MakeSellを通すメリット（セラー側）

| メリット | 内容 |
|----------|------|
| 実績が公開される | 次の案件獲得に有利 |
| 未払いリスクなし | 決済完了後に分配される |
| 契約が明確 | 責任範囲が文書化されてる |
| セラーランク | 実績積むと優先マッチング（将来） |

### 直接取引のリスク（両者に伝える）

| リスク | 内容 |
|--------|------|
| 実績が残らない | 次のマッチングに活かせない |
| トラブル時に証拠なし | 「言った言わない」になる |
| 未払い・持ち逃げ | MakeSellは関与しない |
| 契約不備 | 責任分界が曖昧→揉める |

### 規約への記載文

```
【直接取引について】

MakeSellは、ビルダーとセラーが本プラットフォーム外で
直接連絡・取引を行うことを禁止しません。

ただし、プラットフォーム外での取引については、
MakeSellは一切関与せず、いかなる責任も負いません。

プラットフォーム経由の取引には、以下のメリットがあります：
・売上記録・実績の可視化
・契約条件の明文化
・決済ログによるトラブル防止
・自動分配による請求負担軽減
```

### 将来の拡張案（手数料差別化）

| 取引形態 | MakeSell手数料 | 実績反映 |
|----------|---------------|----------|
| プラットフォーム経由 | 10% | あり |
| 直接取引（申告制） | 0% | なし |

→ **「手数料払ってでも実績残したい」人がプラットフォームを使う構造**

## 16. 分配率設計

### 分配プラン（3段階固定）

| プラン名 | セラー取り分 | ビルダー取り分 | MakeSell | 想定ケース |
|----------|--------------|----------------|----------|------------|
| **ライト** | 20% | 70% | 10% | 紹介程度、既存顧客への案内 |
| **スタンダード** | 30% | 60% | 10% | 通常の営業活動（推奨デフォルト） |
| **プレミアム** | 40% | 50% | 10% | フルコミット、独占販売 |

### 具体例（月額¥5,000のSaaSが売れた場合）

| プラン | セラー | ビルダー | MakeSell |
|--------|--------|----------|----------|
| ライト | ¥1,000 | ¥3,500 | ¥500 |
| スタンダード | ¥1,500 | ¥3,000 | ¥500 |
| プレミアム | ¥2,000 | ¥2,500 | ¥500 |

### プラン選択ガイド（ビルダー向け）

```
【ライト（セラー20%）を選ぶ場合】
・セラーは「紹介するだけ」でOK
・顧客対応・サポートはビルダーが全部やる
・複数セラーに同時依頼したい（非独占）

【スタンダード（セラー30%）を選ぶ場合】
・セラーに積極的な営業活動を期待
・一次問い合わせ対応はセラーに任せたい
・迷ったらこれ（推奨デフォルト）

【プレミアム（セラー40%）を選ぶ場合】
・独占販売権を付与する
・セラーにフルコミットしてほしい
・高単価商品（月額1万円以上）向け
```

### 独占/非独占との組み合わせ推奨

| 販売形態 | 推奨プラン | 備考 |
|----------|------------|------|
| 非独占（複数セラー可） | ライト〜スタンダード | 競争原理が働く |
| 独占（1セラーのみ） | スタンダード〜プレミアム | セラーのコミット確保 |

> **注意：独占 + ライトはNG**（セラーのモチベーションが出ない）

### 継続課金（サブスク）の分配ルール

| 項目 | 設定 |
|------|------|
| **分配方式** | 継続分配（顧客が継続する限り毎月分配） |
| **分配期間** | 無期限（解約まで） |
| **解約時** | 解約月の分配で終了 |

**継続分配を採用する理由：**
- セラーが「売って終わり」にならない
- 顧客フォローへのインセンティブになる
- LTV意識の質の高い営業になる

## 18. 返金リスク対策

### リスクシナリオ

```
顧客が¥10,000を支払い
    ↓
自動分配完了（ビルダー・セラーが即出金）
    ↓
後日、顧客が返金要求
    ↓
ビルダー・セラーの残高ゼロ
    ↓
回収不能リスク発生
```

### 対策①：入金保留期間

| 項目 | 設定 |
|------|------|
| **保留期間** | 7日間 |
| **仕組み** | Stripeの`payout_schedule`で制御 |
| **入金開始** | 決済完了から8日目以降 |

### 対策②：返金ポリシーの制限

**顧客向け返金ポリシー：**

```
【返金ポリシー】

・購入後7日以内：全額返金可能
・購入後8日以降：原則返金不可
・月額課金：当月分の返金不可、次月から解約

※返金申請はMakeSellサポートまでご連絡ください
```

| 商品タイプ | 返金可能期間 | 返金条件 |
|------------|--------------|----------|
| 買い切り | 7日以内 | 未使用の場合 |
| 月額課金 | 原則不可 | 次月から解約対応 |
| 年額課金 | 7日以内 | 未使用の場合のみ全額 |

### 対策③：返金時の負担ルール

| 返金原因 | 負担者 | 備考 |
|----------|--------|------|
| ビルダー起因（バグ、機能不備、サービス停止） | **ビルダー全額** | プロダクト責任 |
| セラー起因（誇大広告、ミスマッチ誘導） | **セラー全額** | 販売責任 |
| 顧客都合（気が変わった等） | **按分負担** | 分配率に応じて |
| 不正利用・チャージバック | **按分負担** | 分配率に応じて |

> **MakeSellは返金費用を負担しない**（プラットフォーム免責）

### 対策④：ネガティブバランス対応

Stripe Connectの自動相殺機能を有効化：

```
セラーの次回売上 ¥5,000
    ↓
前回返金の未回収分 ¥3,000 を自動相殺
    ↓
実際の入金 ¥2,000
```

| 項目 | 設定 |
|------|------|
| **自動相殺** | 有効 |
| **相殺上限** | 次回売上の100%まで |
| **回収不能時** | MakeSellから督促 → 最終的にアカウント停止 |

### 返金フロー図

```
┌─────────────────────────────────────────────────────┐
│  顧客から返金リクエスト                              │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│  返金可否判定                                        │
│  ├→ 7日以内？ → 返金処理へ                          │
│  └→ 7日超過？ → 原則拒否（個別対応）                │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│  返金原因の特定                                      │
│  ├→ ビルダー起因 → ビルダー負担                     │
│  ├→ セラー起因 → セラー負担                         │
│  └→ 顧客都合 → 按分負担                             │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────┐
│  残高から自動控除                                    │
│  ├→ 残高あり → 即時控除・返金実行                   │
│  └→ 残高不足 → 次回売上から相殺                     │
└─────────────────────────────────────────────────────┘
```

### 契約書追記条項

```
【第○条（返金時の負担）】

1. 顧客から返金要求があった場合、以下のルールに従い処理する。

2. 返金原因の負担区分
   (1) ビルダー起因（プロダクトの瑕疵、機能不備、サービス停止等）
       → ビルダーが返金額の全額を負担する
   (2) セラー起因（誇大広告、虚偽説明、不適切な販売行為等）
       → セラーが返金額の全額を負担する
   (3) 顧客都合（単純な気変わり、誤購入等）
       → ビルダー・セラーが本契約の分配率に応じて按分負担する

3. MakeSellは、いかなる場合も返金費用を負担しない。

4. 返金発生時、MakeSellは負担者のStripeアカウント残高から
   返金相当額を自動控除する権利を有する。

5. 残高不足の場合、次回以降の売上から相殺する。
   相殺完了まで、当該アカウントへの入金を保留できる。

6. 相殺による回収が3ヶ月以上不可能な場合、
   MakeSellは当該アカウントを停止し、
   別途請求を行う権利を有する。
```

### 返金率の想定

| 業種 | 一般的な返金率 | MakeSell想定 |
|------|----------------|--------------|
| SaaS（月額） | 1〜3% | 2%以下 |
| デジタル商品 | 3〜5% | - |
| 物販 | 5〜10% | - |

> **SaaS中心のため、返金リスクは限定的と想定**

## 19. 最初の10人戦略（初期指針）

> **注意：** この戦略は初期仮説。実際に動かしながら柔軟に変える前提。

### ターゲット優先度

| 優先度 | 役割 | 誰を狙うか | 理由 |
|:------:|------|-----------|------|
| 1 | ビルダー | 我々自身 + 知人の個人開発者 | 最初の商材は自分たちで用意 |
| 2 | セラー | 知人紹介のSaaS営業経験者 | 価値をすぐ理解してくれる |
| 3 | 両方 | Xで「作ったけど売れない」と言ってる人 | 課題が顕在化している |

### コールドスタートの考え方

```
最初の1マッチングを自分たちで作る
    ↓
成功事例を1つ作る
    ↓
それをXで発信
    ↓
オーガニック流入を待つ
```

> **最初は泥臭くてOK。** スケールは後で考える。

### 「鶏と卵」問題への対応

| 問題 | 対策 |
|------|------|
| ビルダーがいないとセラーが来ない | 我々のプロダクトを最初の商材として登録 |
| セラーがいないとビルダーが来ない | 最初のマッチングは自分たちで仲介。成功事例を作る |
| 両方少ないと魅力がない | 「招待制」で希少性演出。審査通過をステータス化 |

## 21. セラー活動チェック（初期指針）

> **注意：** 検知も対応も自動化。人の介入は最小限に。

### 自動チェックフロー

| 経過日数 | 自動処理 |
|:--------:|----------|
| **14日** | リマインドメール自動送信「販売活動は始まっていますか？」 |
| **30日** | 活動なしの場合、自動で「困っていることはありますか？」メール送信 |
| **45日** | 成果ゼロの場合、ビルダーに自動通知「セラーの活動状況をお知らせします」 |
| **53日** | セラーに最終警告メール自動送信「7日後に契約が自動解除されます」 |
| **60日** | 成果ゼロで自動解除 + 双方に自動通知 |

### 独占契約の場合（厳格運用）

| 経過日数 | 自動処理 |
|:--------:|----------|
| **14日** | 活動報告リマインド自動送信 |
| **30日** | 成果ゼロの場合、ビルダーに「独占解除オプション」を自動提示 |
| **37日** | セラーに警告メール「7日後に独占権が解除されます」 |
| **45日** | 成果ゼロで独占→非独占に自動切替 + 双方に通知 |

### 自動メールテンプレート例

```
【14日目】
件名：販売活動は順調ですか？

○○さん

「{プロダクト名}」のセラーとして承認されてから14日が経ちました。
販売活動は始まっていますか？

困っていることがあれば、このメールに返信してください。

---

【53日目】
件名：【重要】契約自動解除まで7日

○○さん

「{プロダクト名}」のセラー契約から53日が経過しましたが、
売上が記録されていません。

このまま7日間売上がない場合、契約は自動解除されます。

継続を希望する場合は、このメールに返信してください。
```

> **設計思想：**
> - 人が介入しなくても回る仕組み
> - でも「機械的に切られた」と感じさせない文面
> - 返信があれば個別対応の余地を残す

## 22. KPI未達時の学習仮説（初期指針）

> **注意：** 数字に縛られすぎない。初期は「なぜ？」を考える材料として使う。

### 未達時に考えること

| 指標 | 未達時の問い |
|------|-------------|
| ビルダー登録少ない | → 価値が伝わってない？ リーチが足りない？ |
| セラー登録少ない | → 商材の魅力不足？ セラー側メリット不明確？ |
| マッチングしない | → 情報が足りない？ 選択肢が少ない？ |
| 売上出ない | → セラーが動いてない？ プロダクトの問題？ |

### 仮説検証の優先順位

```
売上発生しない
    ↓
まずマッチング成立しているか確認
    ↓
├── マッチングしていない → 供給側（ビルダー）の問題
│   ├── ビルダーが少ない → 集客施策
│   └── ビルダーはいるがマッチしない → プロダクト情報の改善
│
└── マッチングしているが売上ゼロ → 需要側（セラー）の問題
    ├── セラーが動いていない → 活動チェック強化
    └── セラーが動いているが売れない → 販売支援 or プロダクト品質
```

### 撤退判断（ざっくり）

| 期間 | 状態 | 考えること |
|------|------|-----------|
| 3ヶ月 | 何も動かない | 根本から見直す |
| 6ヶ月 | 売上ゼロ | 続けるか判断 |

> **初期の数字は「答え」じゃなく「問い」を生むためのもの。**
> **未達でも学びがあればOK。**
